How to Mock a gRPC Service in CI

Mocking HTTP in CI is a solved problem. Mocking gRPC isn't, because the tooling assumed everyone would just bring up the real service. This is a walkthrough of the actual options — and why we ended up building one of them.

The problem

Your service depends on a gRPC API — an internal one, a vendor's, or a Google/AWS managed service. Your CI needs to run tests that call it. You have three options, and only one of them is good:

  • Run the real service in CI. Doable for internal services, painful for vendor ones, impossible for SaaS. Adds build time and turns flaky-upstream incidents into your CI's problem.
  • Hand-roll a fake. Generate Go/Rust/Python stubs from the proto, implement the methods you care about, run it in a sidecar. Works, but you maintain it forever.
  • Use a generic gRPC mock server. What this post is about.

Step 1: Have the proto file

A gRPC mock needs the same proto your real service uses. If you're calling an internal service, grab the .proto from its repo. For vendor APIs, the proto is usually published — Google's are in googleapis/googleapis, AWS in their respective SDK repos.

Save it somewhere your CI can read it. For this post, assume proto/orders.proto with a service like:

service OrderService {
  rpc GetOrder(GetOrderRequest) returns (Order);
  rpc ListOrders(ListOrdersRequest) returns (stream Order);
}

Step 2: Start the mock server

With MockForge's CLI installed (cargo install mockforge-cli), one command brings up a gRPC server that implements every method in the proto. Each method returns realistic dummy data inferred from the message schema:

$ mockforge serve --proto proto/orders.proto --grpc-port 50051
Listening on 0.0.0.0:50051 (gRPC reflection enabled)

Reflection is on by default, so you can probe with grpcurl localhost:50051 list before you wire it into your client.

Step 3: Point your tests at it

In your test setup, override the gRPC channel to dial localhost:50051 instead of the real upstream. In Go:

conn, err := grpc.Dial(
    os.Getenv("ORDER_SERVICE_URL"), // "localhost:50051" in CI
    grpc.WithTransportCredentials(insecure.NewCredentials()),
)

Use a fixed ORDER_SERVICE_URL env var in your CI workflow, and the tests run against the mock without code changes.

Step 4: Shape the synthetic data

Default behavior is "return a valid response for every method" — the mock infers types from the proto and emits schema-correct dummy data. That covers most CI tests where you just need the call to succeed.

When you need specific field values (a fixed order ID, a known total) the smart-mock generator accepts field-name overrides at startup, so a request for an Order response can be biased toward realistic content per field. Today this lives in code rather than config; the patterns are documented in the gRPC mocking guide, and richer YAML-driven per-method overrides are on the near-term roadmap. For the "does my retry-and-deadline logic work" class of tests, the default-generation behavior is usually enough.

Step 5: Wire it into your CI workflow

For GitHub Actions, that's a single service block:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - run: cargo install mockforge-cli
      - run: |
          mockforge serve --proto proto/orders.proto --grpc-port 50051 &
          sleep 2
      - run: go test ./...
        env:
          ORDER_SERVICE_URL: localhost:50051

If you'd rather not install in CI, MockForge Cloud will host the same mock at <slug>.mocks.mockforge.dev — point your tests at the URL and skip the install step entirely.

When to reach for something else

Two cases where a generic gRPC mock isn't the right tool. First, contract tests — if you need to assert that the real service's responses match its spec, you want Pact or similar, not a mock. Second, performance benchmarks — mocks return synthetic data instantly, which gives you misleading p99 numbers. Use the real service or a recorded replay.

For everything else — feature work that depends on an unfinished service, CI tests that shouldn't hit production, isolated reproducer environments — a proto-driven mock is the cheapest path.
